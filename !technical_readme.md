# Technical README: Folder Map and Workflow Rules

This file is the main navigation guide for empirical research projects initialized from this template.

## LLM Startup Rule

At the start of work, read only the startup Markdown files first:

1. `README.md`
2. `INITIALIZE_PROJECT.md`
3. `!technical_readme.md`
4. `!project_overview.md`
5. `!llm_record.md`
6. `approval_policy.md`

After that, read lower-level README files only when they are relevant to the current task.

## Documentation Layering Principle

Higher-level README files should explain where to find things and which lower-level README to read next.

Folder-specific details belong in the README inside that folder.

Do not duplicate detailed inventories across levels unless duplication is necessary for navigation.

Do not create a new README file unless the human explicitly approves it.

## Example Data Rule

The template includes example documentation files under `examples/data_documentation/`.

These files are examples only. They must not be used as real data.

During initialization, after real raw data are added or linked and documented, the coding LLM must delete `examples/data_documentation/` unless the human explicitly approves retaining it as template documentation only. Record any retention approval in `!llm_record.md`.

## Where to Find Things

| Need | Go to | Notes |
|---|---|---|
| Initialization protocol | `INITIALIZE_PROJECT.md` | Coding LLM must follow this when creating a new project from the template. |
| Project-level research overview | `!project_overview.md` | Project description, research question, mechanism, data, and identification. |
| LLM memory | `!llm_record.md` | Compact durable memory for future LLMs. |
| Approval rules | `approval_policy.md` | Defines actions that require human approval. |
| Durable project inventory | `project_inventory.md` | Lightweight inventory of major components. |
| External dependencies | `external_dependencies.md` | Documents inputs outside the project folder. |
| Open research questions | `open_questions.md` | Backlog for identification, measurement, data, writing, and workflow questions. |
| Configuration scaffold | `config/` | Contains `paths.yaml`; reference-only unless code explicitly consumes it. |
| Code | `code/` | Data processing and empirical analysis scripts. |
| Raw data | `data_raw/` | Original source data; read-only. |
| Intermediate data | `data_intermediate/` | Code-generated checkpoints and diagnostics. |
| Final data | `data_final/` | Analysis-ready datasets generated by code. |
| Results | `result/` | Tables, figures, and review outputs; approval required before LLM modification. |
| Task records | `tasks/` | Work orders, execution notes, review notes, and approval gates. |
| Cross-LLM bridge | `llm_bridge/` | Explicit-trigger communication between research LLM and coding LLM. |
| Skills | `skills/` | Optional-trigger instructions for code review, China administrative matching, and monetary units. |
| Example data documentation | `examples/data_documentation/` | Template example only; delete after initialization unless retained with approval. |

## Skills Rule

The `skills/` folder is intentionally retained because China-focused empirical research often requires special handling of administrative divisions, monetary units, deflation, and silent code bugs.

Do not read every skill by default. Trigger them only when relevant:

- code modification: read `skills/short_silent_bug_checklist.md`;
- comprehensive code audit: read `skills/full_silent_bug_checklist.md`;
- Chinese administrative unit matching or merging: read `skills/china_administrative_zone.md`;
- monetary variables, exchange rates, deflation, logs of monetary values, or real/nominal comparisons: read `skills/monetary_units_deflation_SKILL.md`.

Do not modify anything inside `skills/` without explicit human approval.

## Role Clarification

The human researcher approves substantive changes, code runs, result generation, empirical specification changes, external dependency decisions, and raw-data decisions.

Research-oriented LLMs advise on research design, empirical strategy, documentation, writing, and Codex-ready prompts.

Coding LLMs/Codex implement approved code and documentation tasks. They must not silently change specifications, samples, controls, fixed effects, clustering, estimators, treatment definitions, variable definitions, or result paths.

## Workflow-Ready Status

A project is workflow-ready when documentation, paths, raw-data descriptions, approval boundaries, and folder rules are in place. It is not analysis-ready until validated data pipelines, final datasets, empirical specifications, and result-generation scripts exist.
